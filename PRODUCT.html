Product Overview

Turn information overload into focused action

Technical Deep Dive · Ask AI — Layers 1 & 2

From raw query to structured retrieval plan

When a user types a question, Chief of Staff does not send that text directly to an LLM. It runs a deterministic, multi-stage pipeline that first understands intent, resolves entities, plans what data to fetch, retrieves it in parallel, then ranks and synthesises it. The full pipeline completes in under 2 seconds.

1 Intent Classification gpt-4o-mini · temp 0 3 s timeout · regex fallback

The classifier maps every query to one of 13 named intents, extracts structured entities, infers a temporal anchor, and computes a two-dimensional retrieval weight that controls which downstream path activates.

subscriptions

bills_payments

Classification runs in two passes. Pass 1 is pure regex — named entity triggers (merchant names, people triggers, spending keywords) fire deterministic overrides before any LLM call. Pass 2 calls gpt-4o-mini with a 3-second hard timeout; if it times out, the system falls back to operational_summary at confidence 0.3 rather than blocking.

The classifier also computes a retrieval weight pair — operational_weight and investigative_weight — that governs layer 3. Deterministic overrides hard-set these weights based on intent:

finance / spending_analysis0.11.0Requires exact transaction rows, not summaries
search_lookup0.01.0Pure data retrieval; insights add noise

2 Entity Resolution deterministic · no LLM

Named entities extracted by the classifier are still abstract strings — "Uber", "Sarah", "last month". Entity resolution converts them to concrete database identifiers and date ranges before any query runs.

Entity typeResolution methodOutput
Person name.ilike('%name%') fuzzy match on contacts.name and contacts.email, top 3 resultsArray of contact_id UUIDs
Merchant name100+ alias dictionary with word-boundary matching — "SWIGGY*", "Swiggy food" → canonical swiggyNormalised merchant string
Relative perioddate-fns conversion — last_month → start/end of prior calendar month in UTC{from: ISO, to: ISO}
Event anchorSpecial cases: salary_credit looks up last salary transaction; named_contact_interaction finds last email with that contactDate range window around event

Temporal resolution also emits a temporalConfidence score. If it falls below 0.6, the system appends a clarification note to the final LLM system prompt, telling GPT-4o to acknowledge ambiguity in its answer rather than hallucinate a date.

Retrieval planning, parallel execution, and ranking

3 Retrieval Plan Construction rule-based · budget-capped

The planner maps intent → ordered list of retrieval steps. Each source has a known confidence score, estimated cost in milliseconds, and a maximum result count. Steps are accumulated until the 1 200 ms budget is exhausted.

SourceConfidenceEst. costMax resultsActivated by
aggregated_finance0.9590 ms1any finance / spending intent
sql_transactions0.9580 ms20finance, spending_analysis, bills
sql_commitments0.9060 ms10commitments, productivity
sql_tasks0.9060 ms10commitments, productivity, operational_summary
operational_insights0.8550 ms15operational_weight ≥ 0.3
sql_communications0.85100 ms10search_lookup, relationship, travel, bills
sql_meetings0.8080 ms5scheduling, relationship
vector_search0.55200 ms8investigative_weight > 0.5 or semantic intents

The planner uses two gate conditions: the operational path fires when operational_weight ≥ 0.3 or the intent is a finance type; the investigative path fires when investigative_weight > 0.3 or keyword overrides are present. Both paths can be active simultaneously for mixed queries like "what did I spend on Uber and when did I last email them?"

4 Parallel Data Retrieval Promise.allSettled · partial failure tolerance

All budgeted steps fire simultaneously. Promise.allSettled ensures a failed or slow source (e.g., vector search timing out) never blocks the others. Each result is tagged with its source type so the ranker and LLM know provenance. Notable query mechanics:

SourceKey filtersNotes
sql_communicationscontactIds (IN), emailCategory (IN), dateRange, topics (ILIKE across subject + body_summary)Topic filter is OR-flattened across both columns
sql_transactionsmerchants (IN), transactionTypes (IN), dateRange, amount (eq)Category filter intentionally omitted at query level — applied in ranking
sql_commitmentscontactIds (IN), due_date ≤ range, status ≠ doneOrdered ASC by due_date so overdue items come first
aggregated_financeuser_id, date rangeReturns totals by category, merchant, currency, and weekly trend — one structured object
operational_insightsstatus = active, snoozed_until is null or pastOrdered by priority_score DESC; category filter applied if intent is specific
vector_searchquery embedding vs pgvector HNSW index90-day lookback window, matchCount = max_results

5 Unified Ranking + LLM Response gpt-4o · streaming · multi-turn

All retrieved items — regardless of source type — are normalised into a common schema and scored by a profile-aware weighted ranker. The profile is chosen based on the primary intent. Component weights:

Profile → Intent Keyword match Semantic Entity overlap Recency Src importance Urgency
factual_finance — finance / spending0.350.150.150.250.050.05
semantic_lookup — search_lookup0.100.350.250.150.050.10
relationship — contacts & people0.100.250.350.150.050.10
operational — default / summary0.050.100.100.150.100.50

Per-item urgency is not uniform — it is assigned by source: commitments score 0.8 if overdue / 0.4 otherwise; emails with requires_action=true score 0.6; meetings and transactions score 0.3; vector results score 0.2. Recency decays linearly over 90 days: score = max(0, 1 − daysOld / 90).

After scoring, diversity caps prevent any one source from dominating the context window: max 6 operational insights, 4 emails, 3 transactions, 3 commitments, 2 meetings, 2 vector results. The top 20 items are assembled into a structured evidence block tagged with [TRANSACTION], [EMAIL], [MEETING], [COMMITMENT], [INSIGHT], and [ANALYTICS] markers.

This block, plus the system prompt (role: "proactive AI Chief of Staff"), is passed to GPT-4o with streaming. The response header X-Assistant-Context carries the conversation state — active entities, temporal anchor, sources used — back to the client for multi-turn continuity.

Design rationale: Pure RAG over embeddings would conflate "how much did I spend on Uber?" with calendar events mentioning Uber. The intent + entity layers route each query to the correct table. The ranking layer then ensures SQL rows, vector matches, and precomputed insights are balanced — not whichever source returned the most rows.

Technical Deep Dive · Inbound Data Pipeline

How emails and meetings become structured intelligence

Every piece of information in Chief of Staff flows through a background worker pipeline. Nothing is processed synchronously — webhooks enqueue BullMQ jobs, and workers run in a separate Railway process with their own env loading and service-role Supabase client (bypassing RLS).

Email processing — 5-stage worker pipeline

1

Fetch & normalise process-email.ts

Gmail API fetches the full message. Headers extracted (from, to, subject, date, threadId, list-unsubscribe). Sender email and name parsed and upserted to the contacts table.

Importance heuristic seeds an importance_score starting at 0.5: +0.2 if Gmail IMPORTANT label and no list-unsubscribe; +0.1 if not CC'd; +0.15 for urgency keywords (urgent, asap, deadline, invoice, contract, term sheet); −0.45 if unsubscribe present. Score clamped to [0, 1].

2

AI triage gpt-4o-mini · max_tokens 700

Calls triageEmail() with business context, sender info, and body. Returns a Zod-validated JSON object with: summary, sentiment (positive / neutral / negative / urgent), refined importance_score, requires_action, action_description, email_category (one of 22), fallback_category, and confidence.

Categories are prioritised in a fixed order: Travel > Account Security > Finance Bills > Transactions > Subscriptions > Receipts > Legal > Calendar > Shipping > Productivity Tools > Career > Social > … > Other.

3

Financial extraction conditional · gpt-4o-mini

Gate: runs only if the email category is a finance type. Extracts amount, currency, merchant, transaction_type, and confidence. Sets a needs_review flag when the triage category disagrees with the extraction result or confidence is below 0.5.

If review is not needed, calls runDedup() to normalise against existing transactions (same merchant + similar amount ± 2% within 3 days = duplicate). The merchant name and category are normalised via the same 100+ alias dictionary used in entity resolution.

4

Commitment & task extraction gpt-4o · if requires_action

If requires_action = true and AI confidence ≥ 0.6, calls extractCommitments() to find promises in the email body — who committed, to whom, by when. Each commitment is stored with source_type = "email".

If the capped importance score ≥ 0.7, a task is also created (priority = "high" if score ≥ 0.85, else "medium") with the AI's action description as the title.

5

Embed & trigger operational state Voyage AI · 30 s delay

The email summary is chunked and embedded using Voyage AI voyage-3-lite (1 024-dim). The embedding is stored in the communications.embedding vector column and indexed via pgvector HNSW for sub-100 ms semantic search.

Finally, a compute-operational-state job is enqueued with a 30-second delay and a deduplication key — ensuring batch updates don't trigger redundant recomputes.

Meeting processing pipeline summarize-meeting.ts

Triggered when a Zoom meeting ends. The worker downloads the VTT transcript, parses speaker turns, and calls summarizeMeeting() with the title, attendee list, transcript, and business context. GPT-4o returns a structured JSON containing:

Executive summary

3–5 sentence narrative of what was discussed and decided

Key decisions

Each decision with full context — not just the conclusion

Action items

Owner, due date, and description per item. Items owned by the founder auto-create tasks.

Commitments

Promises extracted with due date and confidence ≥ 0.6 threshold

Follow-up emails

Suggested recipients and topics for post-meeting outreach

Sentiment

productive / tense / unclear / routine — surfaced in the meetings list view

The summary, decisions, and action items are concatenated and embedded (Voyage AI) to memory — making meeting content searchable from Ask AI using semantic queries like "what was decided about pricing in Q1?"

System Architecture

How data flows through the platform

Chief of Staff is a real-time event-driven system. Every email and meeting update arrives via webhook, is processed asynchronously by a background worker, and the results are immediately available in the UI.

Gmail / Calendar / Zoom webhook

AI triage (GPT-4o-mini)
→
Embed → pgvector

Operational insights

OAuth tokens for every integration are stored AES-256-GCM encrypted. Row-Level Security in Postgres ensures users never see each other's data — enforced at the database layer, not just in application code. The BullMQ worker runs as a separate Railway process using a service-role Supabase client (bypasses RLS) and lazy-initialises all Redis and Supabase connections to avoid module-evaluation errors before env vars load.

Opportunity

The path forward

The executive assistant software market is large and fragmented. Chief of Staff occupies a defensible position: the only tool that combines email intelligence, meeting intelligence, and financial intelligence into a unified, queryable layer. As the platform accumulates longitudinal data per user, the Ask AI feature compounds in value — making it increasingly difficult for single-purpose tools to match the contextual depth of answers Chief of Staff can provide.

The moat is data depth, not features. Every email processed, every commitment extracted, every transaction categorised makes future answers more accurate and more personal. The longer a user is on the platform, the better it understands their professional context — relationships, recurring obligations, spending patterns, communication style.
